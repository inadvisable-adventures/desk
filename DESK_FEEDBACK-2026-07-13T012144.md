# Desk feedback: widgets/hex_flower renders a blank page

Written per TODO `4ab5875`, in response to a report that
`widgets/hex_flower` (a port of `../../claude-projects/hexsheet`'s
hex-flower sheet item, built by another Claude instance running inside
a Desk) comes up blank when the Desk is actually run. This is an
investigation + feedback document only -- no source was changed as
part of this item, including `widgets/hex_flower` itself.

## Diagnosis

`hex_flower` is a `kind: "html"` widget whose entry document
(`index.html`) loads its actual logic the normal, unremarkable way any
web project would: a compiled TypeScript bundle and a couple of CSS
files, referenced via ordinary relative paths --
`<script type="module" src="dist/main.js">`, `<link rel="stylesheet"
href="dice-pool.css">`, and so on. The build itself is fine (`dist/`
contains real, correctly-compiled output); the custom element
(`<hex-flower>`) and its `customElements.define(...)` call are present
and correct. The widget doesn't reference `window.desk`, `fetch`, or
anything else unusual in a way that would explain the failure from
reading the code alone.

Running it for real (a headless `QWebEnginePage`/the real
`ChromiumWidget`, with console-message capture and a post-load DOM
check) shows: zero console messages, and the `<hex-flower>` element's
`shadowRoot` is `null` -- meaning `connectedCallback` never ran, which
means the module script graph rooted at `dist/main.js` never finished
evaluating. Nothing in `hex_flower` itself explains that; a minimal,
freshly-written reproduction widget (a single external `<script src>`,
otherwise trivial) reproduces the identical failure, independent of
anything specific to `hex_flower`'s code.

The actual mechanism, confirmed directly against the running Local Web
Server (not inferred): a widget's main page is loaded with the
per-launch auth token as a query parameter
(`.../widgets/hex_flower/index.html?token=...`), which
`TokenAuthMiddleware` requires on every request. Fetching
`dist/main.js` (or any relatively-referenced sibling asset) *without*
that token returns `401 Unauthorized`; with it, `200`. A plain
relative-path browser reference in `<script src>`/`<link href>`
resolves against the *path* of the current document, but does **not**
carry the current document's query string forward -- that's ordinary
URL-resolution behavior, not a bug in the browser. And unlike
`fetch()`/`XMLHttpRequest` (which is exactly how the injected Bridge
client, `desk.server.bridge_client`, makes its own calls), a native
`<script src>`/`<link href>` tag has no way to attach a custom header
either. So every sub-resource request a `kind: "html"` widget makes via
an ordinary relative reference arrives at the server with no
credential at all, and is rejected before the browser ever gets the
file -- which, for a `<script type="module">`, silently aborts the
whole module graph with no visible error (this is why the console was
empty: the browser's own module loader reports failed sub-resource
fetches to the network layer, not to `console`, and nothing in Desk
surfaces network-level failures back to the widget or the user).

This is a general gap, not specific to `hex_flower`: **any**
`kind: "html"` widget that is more than a single self-contained HTML
file -- i.e. any ordinary multi-file web project built the normal way
-- will silently fail to load its own assets. The only reason existing
tempui `DefineWidget` widgets don't hit this is that `DefineWidget`
requires a single base64-encoded HTML document with everything inlined
(TODO `91b3f42`) -- a constraint that happens to sidestep this bug as
a side effect, not because anyone identified and intentionally avoided
it.

A second, smaller, independent finding: `hex_flower`'s state layer is
ported directly from `hexsheet`'s own architecture -- it dispatches
bubbling `RegisterSignalSender`/`RegisterDatastore` `CustomEvent`s
expecting a "sheet" parent controller (from the original project) to
answer them. Desk has no such controller; those events are simply
never handled. Independent of the blank-page bug above, this widget's
persistence would not work in Desk even once assets load, because it
was never wired to Desk's actual Bridge API
(`self.getLocalStorage`/`setLocalStorage`, from TODO `5734529`).

## (a) Documentation suggestions

- `design-docs/architecture.md`'s description of `kind: "html"`
  widgets doesn't mention the auth token at all. It should state
  plainly that every request the browser makes for a widget's own
  page -- not just the top-level navigation -- must carry the
  per-launch token, and that ordinary relative-path resource
  references (`<script src>`, `<link href>`, CSS `url(...)`, `<img
  src>`, etc.) do **not** carry the token forward and will be
  rejected, so a widget with more than one file will not load until
  the underlying gap (see (b) below) is fixed.
- More generally, there's no single place that lays out, for someone
  building a `kind: "html"` widget from scratch (as opposed to a
  tempui `DefineWidget` single-file widget), what does and doesn't
  work yet -- e.g. "single self-contained HTML file: works"; "multiple
  files loaded via relative paths: currently broken, see [gap]".
  Without that, a reasonable, competently-built ordinary web project
  (exactly what happened here) silently fails with zero diagnostic
  signal, and there's no way for whoever built it to have known in
  advance.
- Nothing currently tells a widget author *how to tell* their widget
  failed to load. A "how to debug a blank widget" note (start from:
  is the custom element even defined? check
  `document.querySelector(...).shadowRoot`; are there failed network
  requests?) would have shortened this investigation considerably --
  right now the failure mode is a silent blank page with no console
  output at all, which looks identical to several other possible
  failures (bad HTML, crashed script, wrong entry point, etc.).

## (b) tempui / widget-feature improvement suggestions

- **Fix the root cause.** The clean fix is a same-origin cookie set
  when a widget's main page is served (alongside the existing
  query-param/`X-Desk-Token`-header checks in `TokenAuthMiddleware`,
  not replacing them), since cookies are the one credential mechanism
  the browser attaches automatically to *every* same-origin request,
  including plain `<script src>`/`<link href>` sub-resource loads.
  This is the single highest-leverage fix here: it would make
  ordinary, multi-file `kind: "html"` widgets (built the way any web
  developer would normally build one) work without any special
  casing, and would remove the only reason `DefineWidget` widgets are
  currently forced into single-file base64-inlined form.
- Once that's fixed, it's worth reconsidering whether `DefineWidget`
  still needs to require one inlined file, or whether it could support
  a small set of named files (e.g. an HTML entry plus a script and a
  stylesheet) -- inlining is convenient for an agent to author quickly,
  but it stops working well for anything non-trivial (as `hex_flower`
  itself illustrates, ported from a real multi-file TypeScript
  project).
- Some visible surface for `kind: "html"` widget failures would help
  independent of the fix above -- e.g. surfacing the widget's own
  `javaScriptConsoleMessage` output somewhere inspectable (a log file,
  a debug panel), and/or a generic "this widget failed to render"
  placeholder shown in place of a silent blank page, so a failure at
  least *looks* like a failure instead of looking like a working-but-
  empty widget.
- A known-good, minimal multi-file `kind: "html"` widget example
  (checked in, or referenced from the docs) would give a future Claude
  instance building something like `hex_flower` a template to diff
  against, rather than discovering the relative-path/token gap the
  hard way after already doing a real port of an existing project.
- Separately from the blank-page bug: whatever guidance eventually
  covers the Bridge API's local-storage calls should make clear that a
  ported widget's *own* prior persistence mechanism (custom events,
  global variables, whatever the original project used) needs to be
  explicitly re-wired to `self.getLocalStorage`/`setLocalStorage` --
  porting a widget doesn't make its old persistence approach work
  inside Desk automatically, and there's currently nothing prompting
  an agent to check for that mismatch.
