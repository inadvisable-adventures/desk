# Fix New Desk flow segfault + consolidate into one dialog (COMPLETED)

TODO `4716585`.

## Summary

Reported crash: creating a new Desk, the app segfaulted right as the
user clicked "Yes" on the ".desk_temp" confirmation dialog (a
`QMessageBox`), with the new Desk's seeded Markdown-with-README widget
(TODO `cb2790d`) appearing to show up "at the same time." The full
crash report (kept out of the repo per explicit instruction — not
stored in any project directory) shows a `SIGSEGV`/`EXC_BAD_ACCESS` on
the *main* thread inside `QAbstractItemView::mouseReleaseEvent` ->
`QListView::mouseReleaseEvent` -> `sipQListWidget::mouseReleaseEvent`,
reached via the normal Cocoa event-delivery path
(`QWindowSystemInterface` -> `QApplication::notify` ->
`QAbstractItemView::viewportEvent`). The faulting address looks like
reused/garbage memory (ASCII-digit byte pattern), not a null pointer —
the shape of a use-after-free, not a null-deref.

## Diagnosis (not confirmed by reproducing -- the crash isn't reliably
reproducible on demand, per usual for a use-after-free race)

The "New Desk" flow (`_on_new_desk_requested`) is triggered by
`_DeskListPopup`'s "New Desk" row (`desk_picker.py`) — a
`QListWidget`-based, `Qt.WindowType.Popup`, `WA_DeleteOnClose` widget.
Its own `_activate_item` already closes itself *before* emitting
`new_desk_requested` (a prior fix, `plans/fix-desk-list-popup-deleted
-mid-callback.md`, for a *different* crash: calling `self.close()`
*after* an emit that itself shows a modal dialog used to run `close()`
against an already-deleted `self`). But `close()` with
`WA_DeleteOnClose` only *schedules* a `DeferredDelete` event — it does
not destroy the popup's C++ objects (including its internal
`QListWidget`) synchronously. That deferred deletion actually runs the
next time Qt processes deferred-delete events, which happens inside
*any* nested event loop — e.g. the very next modal dialog's own
`exec()`.

The old "New Desk" flow ran through **up to five** sequential modal
dialogs after the popup closed: name (`QInputDialog`), directory
(`QFileDialog`), an optional development-process.md copy confirm
(`QMessageBox`), a `.desk_temp` creation confirm (`QMessageBox`), and a
`.gitignore` confirm (`QMessageBox`) — each one's `exec()` is a nested
event loop that could be the moment the popup's deferred deletion
actually runs. If a mouse-release event generated by the *same physical
click* that chose "New Desk" was still in flight (queued, not yet
delivered — press and release can be split across event-processing
boundaries) at the moment one of these nested loops processes the
popup's deferred deletion, delivering that stale event afterward hits
an already-destroyed `QListWidget` — a use-after-free, matching the
reported crash's shape and location exactly.

Separately (part 2 of the fix, below): the new-Desk widget-seeding
(TODO `cb2790d`) currently runs *before* `_provision_temp_ui`, so a
just-placed Markdown/Scratch widget is already alive and doing its own
work (file watching, rendering) while a confirmation dialog for a
completely different concern is still up — genuinely "conflicting
things in parallel," per the user's own report and hypothesis, even if
not the literal cause of *this* crash.

## Fix, in three parts (as directed)

### 1. One consolidated `NewDeskDialog`, not five sequential popups

New `src/desk/shell/new_desk_dialog.py`: `NewDeskDialog` (a
`Qt.WindowType.Tool`, non-modal-but-blocking-app-flow floating dialog,
matching `_ItemDialog`'s existing shape rather than introducing a new
pattern) with:

- **(a) Name** — `QLineEdit`.
- **(b) Path** — a read-only `QLineEdit` showing the currently-selected
  directory (defaults to the current Desk's directory, matching the
  old flow's own default), plus a "Browse…" button that opens
  `QFileDialog.getExistingDirectory` and updates the display. This is
  the one nested dialog that can still appear — but only on explicit
  request, embedded *within* the single new-Desk dialog's own already
  -open lifetime, not chained sequentially after it.
- **(c) "Create `.desk_temp` for tempui"** — checkbox, checked by
  default (matches old default behavior of asking-and-usually-agreeing).
- **(d) "Create/update `.gitignore` with Desk-specific patterns"** —
  checkbox, checked by default.
- **(e) "Copy `development-process.md` from the current Desk"** —
  checkbox, shown only if the current Desk actually has one to copy
  from (mirrors the old flow's own `source.is_file()` gate), unchecked
  by default (matches the old flow only offering, never assuming, this
  one). Not explicitly listed in the user's lettered request, but it's
  exactly the same kind of "question popped up during New Desk"
  this whole refactor is about — folding it into the same dialog is
  the same fix applied consistently, not a separate decision.
- **Create / Cancel** buttons.

This collapses what was up to 5 sequential `exec()` calls (name,
directory, dev-process confirm, `.desk_temp` confirm, `.gitignore`
confirm) into at most 2 (`NewDeskDialog` itself, plus an optional
"Browse…" for the path) — directly shrinking the window in which the
`_DeskListPopup`'s deferred deletion could collide with an in-flight
stale mouse event.

### 2. Provision before placing any widget, not after

`DeskWindow.switch_desk` reordered: `current_context
.set_current_desk_directory(...)` and `_provision_temp_ui(...)` (now
optionally given pre-decided answers, see below, instead of always
prompting) both move to run *before* `_load_desk_widgets` — a widget
placed during `_load_desk_widgets` (e.g. `cb2790d`'s seeded Markdown
-with-README) now only ever starts running once every directory/file
-level decision for this Desk has already been made and acted on, not
concurrently with it. This also incidentally fixes a latent,
previously-undiagnosed ordering bug: `current_context`'s directory
used to update *after* `_load_desk_widgets` in `switch_desk` (unlike
`DeskWindow.__init__`, which already documents *why* it must come
first) — any widget reading it synchronously at construction time
could see the *previous* Desk's directory. The picker's own visual
refresh (`_refresh_picker`, MRU list included) stays at the end, after
`add_to_mru` — that's cosmetic, not a construction-time dependency.

`_provision_temp_ui` takes an optional pre-decided-answers parameter
(a small `NewDeskProvisioning` dataclass: `create_temp_ui`,
`create_gitignore`) — when given (only by `new_desk`, fed from the new
dialog's checkboxes), it skips `_confirm_fn`'s dialogs entirely and
uses the checkbox answers directly; when omitted (the ordinary
switch-Desk/change-directory paths), behavior is unchanged (confirm
dialogs, as before) — those paths have no equivalent upfront dialog to
source pre-decided answers from, and in practice `.desk_temp` almost
always already exists for an existing Desk being switched to, so those
dialogs essentially never fire there anyway.

### 3. Re-check existence immediately before creating, abort recoverably if it's already there

Applied to every directory/file creation in the New Desk flow
specifically (the actual site of this report, not a sweep of the whole
codebase — see the Parking Lot item below for that):

- `TempUiManager.provision`: re-checks `temp_dir.is_dir()` immediately
  before `mkdir()`, *after* `ask_create_dir()` returns — that callable
  can pump a modal dialog's own nested event loop for an arbitrary
  amount of time, during which the directory could already have been
  created by something else. If it's already there by the time this
  runs, there's nothing left to do (not an error) — `mkdir(parents=True,
  exist_ok=True)` was already idempotent at the OS level, but this
  makes the "check, then act" intent explicit in the code rather than
  relying on that incidentally.
- `ensure_gitignore_entry`: re-*reads* `.gitignore` (not just a
  boolean re-check) immediately before writing, after `ask()` returns,
  for the same reason — a blind, already-stale in-memory copy could
  otherwise clobber a real concurrent change.
- `new_desk`: re-checks `path.exists()` (the target `.desk` file)
  immediately before `save_current_desk()` actually writes it — by
  that point `switch_desk` above has already done real, non-trivial
  work (provisioning, placing widgets), giving a real window for
  someone else to have created a file at that exact path in the
  meantime. Recoverable: nothing has been written to disk yet at that
  point, so aborting here just means the in-memory session doesn't
  save over it — the same warning dialog as the too-early check uses.
- `_seed_development_process`: already re-checks
  `destination.exists()` immediately before its own `write_text` call
  (no other operation in between) — confirmed correct as-is, no change
  needed.

Also added a `PARKINGLOT.md` item: consider a generalized, reusable
mechanism/pattern for "check immediately before create, abort
recoverably if it's already there" instead of hand-writing this same
shape at each call site — this change only applies it to the New Desk
flow's own creation sites, not every file-creation path in the
codebase (e.g. the Markdown widget's "Save As" already gets equivalent
protection for free from the native macOS save panel's own overwrite
prompt, and Stack's "Save as Markdown" is an intentional, expected
overwrite of a file the user is actively working with, not a
"this shouldn't already exist" case).

## `.gitignore` formatting

`ensure_gitignore_entry` now always writes a blank line, then a
`# Desk-specific` comment, then the entry — for *both* the
create-from-nothing and append-to-existing cases, per explicit
instruction ("in both the cases of create and update"). Applied
uniformly (no special-casing an empty starting file), so a brand-new
`.gitignore` created by this path starts with one blank line before
the comment — a minor, deliberate stylistic quirk, not an oversight.
"Be able to append if it can't already" was checked against the
existing implementation: `read_text()`-then-`write_text()` already
preserves any prior content correctly (creates fresh if missing,
appends after existing content otherwise) — no functional gap found
there beyond the missing blank-line/comment formatting this change
adds.

## Verification

Headless (`QT_QPA_PLATFORM=offscreen`, real `QApplication`, real
files):

- `NewDeskDialog`: default path shown, Browse… updates it, checkboxes
  default correctly (dev-process checkbox hidden/unchecked when there's
  no source to copy from), Create emits the collected answers, Cancel
  emits nothing.
- `ensure_gitignore_entry`: fresh file gets `\n# Desk-specific\n
  .desk_temp/\n`; a file with existing content (with and without a
  trailing newline) gets exactly one blank line before the same
  block; an already-present entry is left alone (no rewrite, `ask()`
  never called); confirms `ask()` not called a second time and the
  file isn't touched if the entry appears between the first check and
  the (simulated) re-check.
- `TempUiManager.provision`: re-check-before-mkdir path exercised by
  creating the directory *during* the `ask_create_dir` callable itself
  (simulating a concurrent creator) — confirms no exception and
  correct end state either way.
- `DeskWindow.new_desk`/`switch_desk`/`_provision_temp_ui` (unbound
  methods on a fake double, the established pattern for
  `DeskWindow`-dependent logic): `current_context`'s directory is set,
  and `_provision_temp_ui` runs, *before* `_load_desk_widgets`;
  pre-decided `NewDeskProvisioning` answers skip confirm dialogs
  entirely; the late re-check in `new_desk` aborts (warns, doesn't
  call `save_current_desk`) if the target path exists by the time it
  runs.
